# Engineering Audit 001

# Flutter Project Structure

---

## Purpose

This audit explains the purpose of every major file and folder generated by `flutter create`.

The objective is to understand the default Flutter project before making architectural changes.

VTOBForge engineering follows the principle:

> Never modify what you do not yet understand.

---

# The Five Engineering Questions

## 1. What is it?

A Flutter project is a complete cross-platform application generated by the Flutter SDK.

Although most application code lives inside the `lib/` directory, Flutter also creates platform-specific projects and configuration files that work together to build the final application.

---

## 2. Why does it exist?

Flutter generates a complete project because every supported platform still requires native build configuration.

Instead of manually creating Android, iOS, macOS, Windows, Linux and Web projects, Flutter creates them automatically and keeps them synchronized.

---

## 3. Who uses it?

The generated project is used by:

- Flutter SDK
- Dart SDK
- Android Gradle
- Xcode
- Browser build tools
- Desktop build systems
- IDEs (VS Code / IntelliJ)
- Continuous Integration systems

---

## 4. When do we modify it?

Some files are modified frequently.

Examples:

- `lib/`
- `pubspec.yaml`
- `README.md`

Other files are modified only when platform-specific configuration is required.

Examples:

- Android permissions
- Firebase configuration
- App signing
- Package names
- Native integrations

---

## 5. Common Mistakes

Common beginner mistakes include:

- Editing generated platform files without understanding them.
- Deleting folders because they appear unused.
- Mixing application code with platform configuration.
- Treating every folder as equally important.

# Flutter Project Structure

## android/

Contains the native Android project.

Responsible for:

- APK generation
- App Bundle generation
- Android permissions
- Signing configuration
- Native Android integrations

---

## ios/

Contains the native iOS project.

Used for iPhone and iPad builds.

---

## macos/

Contains the native macOS desktop project.

---

## windows/

Contains the native Windows desktop project.

---

## linux/

Contains the native Linux desktop project.

---

## web/

Contains the Flutter Web project.

---

## lib/

Contains the application source code.

This is where the majority of engineering work takes place.

---

## test/

Contains automated tests.

---

## pubspec.yaml

Defines the Flutter project.

Responsible for:

- Dependencies
- Assets
- Fonts
- SDK constraints
- Version numbers

---

## pubspec.lock

Stores the exact versions of installed packages.

It should not be edited manually.

---

## analysis_options.yaml

Defines static analysis and linting rules.

Helps maintain consistent code quality.

---

## README.md

Introduces the repository to developers.

---

# VTOBForge Engineering Standards

- Understand every generated file before modifying it.
- Keep platform-specific code inside its platform folder.
- Keep application code inside `lib/`.
- Do not delete generated folders without understanding their purpose.
- Prefer intentional architecture over copied templates.

---

# Summary

A Flutter project is more than a `lib/` directory.

It is a complete multi-platform software project.

Understanding the generated structure allows engineers to make informed architectural decisions instead of relying on trial and error.
